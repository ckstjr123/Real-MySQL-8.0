## 10.3 실행 계획 분석

### 10.3.1 id 컬럼
- 실행 계획에서 가장 왼쪽에 표시되는 id 컬럼은 **단위 SELECT** 쿼리별로 부여되는 식별자 값  
  하나의 SELECT 문장은 다시 1개 이상의 하위(SUB) SELECT 문장을 포함할 수 있음
- 하나의 SELECT 문장 안에서 여러개의 테이블을 조인하면 조인되는 테이블 개수만큼 실행 계획 레코드가 출력되지만 같은 id 값이 부여됨
- 실행 계획의 id 컬럼이 테이블 접근 순서를 의미하진 않음

### 10.3.2 select_type 컬럼
> 각 단위 SELECT 쿼리가 어떤 타입의 쿼리인지 표시되는 컬럼

#### 10.3.2.1 SIMPLE
- UNION이나 서브쿼리를 사용하지 않는 단순한 SELECT 쿼리인 경우 해당 쿼리 문장의 select_type은 SIMPLE로 표시됨(쿼리에 조인이 포함된 경우도 마찬가지)

#### 10.3.2.2 PRIMARY
- UNION이나 서브쿼리를 가지는 SELECT 쿼리의 실행 계획에서 가장 바깥쪽(OUTER)에 있는 단위 쿼리는 select_type이 PRIMARY로 표시됨
- SIMPLE과 마찬가지로 select_type이 PRIMARY인 단위 SELECT 쿼리는 하나만 존재

#### 10.3.2.3 UNION
- UNION으로 결합하는 단위 SELECT 쿼리 가운데 첫번째를 제외한 두번째 이후 단위 SELECT 쿼리의 select_type은 UNION으로 표시됨

#### 10.3.2.4 DEPENDENT UNION
- UNION/UNION ALL로 집합을 결합하는 쿼리에서 표시되며 DEPENDENT는 UNION/UNION ALL로 결합된 단위 쿼리가 외부 쿼리에 의해 영향을 받는 것을 의미

#### 10.3.2.5 UNION RESULT
- UNION 결과를 담아두는 임시 테이블
- MySQL 8.0 버전부터 UNION ALL은 임시 테이블을 사용하지 않음

#### 10.3.2.6 SUBQUERY
- select_type의 SUBQUERY는 **FROM 절 이외에서 사용되는 서브쿼리만**을 의미

#### 10.3.2.7 DEPENDENT SUBQUERY
- **바깥쪽(Outer) SELECT 쿼리의 컬럼에 의존**적인 서브쿼리
- DEPENDENT UNION과 같이 DEPENDENT SUBQUERY 또한 외부 쿼리가 먼저 수행된 후  
  서브쿼리가 실행돼야 하므로 일반 서브쿼리보단 처리 속도가 느릴 때가 많음

#### 10.3.2.8 DERIVED
- **FROM 절에 사용된 서브쿼리**는 select_type이 DERIVED로 표시됨
- DERIVED인 경우에 메모리나 디스크에 생성되는 임시 테이블을 MySQL에서는 '파생 테이블'이라고 함
- MySQL 8.0 버전부터는 FROM 절의 서브쿼리에 대한 최적화도 많이 개선되어 가능하다면 불필요한 서브쿼리는 조인으로 쿼리를 재작성해서 처리함

#### 10.3.2.9 DEPENDENT DERIVED
- select_type 컬럼의 DEPENDENT DERIVED는 해당 테이블이 래터럴 조인으로 사용된 것을 의미
- MySQL 8.0 버전부터 래터럴 조인(LATERAL JOIN) 기능이 추가되면서 FROM 절의 서브쿼리에서도 외부 컬럼을 참조할 수 있게 됨
- 래터럴 조인은 LATERAL 키워드를 사용해야 하며 LATERAL 키워드가 없는 서브쿼리에서 외부 컬럼을 참조하면 오류 발생

#### 10.3.2.10 UNCACHEABLE SUBQUERY
- 서브쿼리에 포함된 요소(사용자 변수 등)에 의해 서브쿼리 캐시가 불가능한 경우 select_type이 UNCACHEABLE SUBQUERY로 표시됨
- SUBQUERY와 DEPENDENT SUBQUERY 캐시 사용 방법 비교
  - SUBQUERY는 외부 쿼리의 영향을 받지 않으므로 처음 한번만 실행해서 그 결과를 캐시
  - DEPENDENT SUBQUERY는 의존하는 외부 쿼리 컬럼  값 단위로 캐시

#### 10.3.2.11 UNCACHEABLE UNION
- UNCACHEABLE과 UNION 두 키워드의 속성이 혼합된 select_type

#### 10.3.2.12 MATERIALIZED
- 서브쿼리 부분이 임시 테이블로 구체화되었음을 나타냄
- 주로 FROM 절이나 IN(subquery) 형태의 서브쿼리 최적화에 사용됨

### 10.3.3 table 컬럼
- MySQL 서버의 실행 계획은 테이블 기준으로 표시되며 테이블에 별칭이 부여된 경우에는 별칭이 표시됨
- <subquery N>, <derived N> 또는 <union M,N>과 같이 '<>'로 둘러싸인 테이블명은 임시 테이블을 뜻하며 '<>' 안에 표시되는 숫자는 단위 SELECT 쿼리의 id 값을 지칭

### 10.3.4 partitions 컬럼
- MySQL 8.0 버전부터는 EXPLAIN 명령으로 파티션 관련 실행 계획까지 모두 확인할 수 있게됨
- 파티션을 참조하는 쿼리의 경우 옵티마이저가 쿼리 처리를 위해 필요한 파티션들만 모아서 실행 계획 partitions 컬럼에 표시해 줌
- 파티션이 여러 개인 테이블에서 불필요한 파티션을 제외하고 접근해야 할 것으로 판단되는 테이블만 골라내는 과정을 파티션 프루닝(Partition pruning)이라고 함
- MySQL을 포함한 대부분의 RDBMS에서 지원하는 파티션은 물리적으로 개별 테이블처럼 별도의 저장 공간을 가짐

### 10.3.5 type 컬럼
- 실행 계획에서 type 이후의 컬럼은 MySQL 서버가 각 테이블의 레코드를 어떤 방식으로 읽었는지를 나타냄
- type 컬럼 값은 각 테이블 접근 방법(Access type)으로 해석하면 됨
- 실행 계획 type 컬럼에 표시될 수 있는 값
    - ~~system~~
    - const
    - eq_ref
    - ref
    - fulltext
    - ref_or_null
    - unique_subquery
    - index_subquery
    - range
    - index_merge(index_merge를 제외한 나머지 접근 방법은 하나의 인덱스만 사용)
    - index
    - ALL(ALL을 제외한 나머지는 모두 인덱스를 사용하는 접근 방법)

#### 10.3.5.2 const
> 테이블 레코드 건수와 관계없이 쿼리가 프라이머리 키나 유니크 키 컬럼을 이용하는  
  where 조건절을 갖고 있으며, 반드시 1건을 반환하는 쿼리 처리 방식
- 다중 컬럼으로 구성된 PK나 유니크 키 중에서 인덱스의 일부 컬럼만 조건으로 사용할 때는 const 타입의 접근 방법을 사용하지 못함
- MySQL 옵티마이저가 쿼리를 최적화하는 단계에서 쿼리를 먼저 실행하여 통째로 상수화하기 때문에  
  실행 계획의 type 컬럼 값이 const(상수)로 표시되는 것

#### 10.3.5.3 eq_ref
- 여러 테이블이 조인되는 쿼리 실행 계획에 한해서 두번째 이후 읽는 테이블의 type 컬럼에 eq_ref가 표시됨
- 조인에서 처음 읽은 테이블의 컬럼 값을 그다음 읽어야 할 테이블의 PK나 유니크 키 컬럼 검색 조건에 사용할 때를 가리킴
  - 두번째 이후 읽히는 테이블을 유니크 키로 검색할 때 그 유니크 인덱스는 NOT NULL이어야 하며,  
    다중 컬럼으로 만들어진 PK나 유니크 인덱스라면 인덱스의 모든 컬럼이 비교 조건에 사용돼야만 eq_ref 접근 방법이 사용됨

#### 10.3.5.4 ref
- 인덱스 종류와 상관없이 동등(Equal) 조건으로 검색할 때는 ref 접근 방법이 사용됨
  ```sql
  SELECT * FROM dept_emp WHERE dept_no = 'd005';
- eq_ref와는 다르게 조인 순서와도 무관함
- 반환되는 레코드가 반드시 1건이라는 보장이 없으므로 const나 eq_ref보단 덜 빠름

#### 10.3.5.5 fulltext
- MySQL 서버의 전문 검색(Full-text Search) 인덱스를 사용해 레코드를 읽는 접근 방법

#### 10.3.5.6 ref_or_null
- ref 방식 또는 NULL 비교(IS NULL) 접근 방법을 의미
  ```sql
  SELECT * FROM titles WHERE to_date = '1985-03-01' OR to_date IS NULL;

#### 10.3.5.7 unique_subquery
- WHERE 조건 절에서 사용될 수 있는 IN(subquery) 형태의 쿼리를 위한 접근 방법으로서 서브쿼리에서 중복되지 않는 유니크한 값만 반환할 때 사용됨

#### 10.3.5.8 index_subquery
- IN 연산자 특성상 IN(subquery) 또는 IN(상수 나열) 형태의 조건은 괄호 안에 있는 값 목록에서 중복된 값이 먼저 제거돼야 하는데, IN(subquery)에서 서브쿼리 결과의 중복 값을 인덱스를 이용해서 제거할 수 있을 때 index_subquery 접근 방법이 사용됨

#### 10.3.5.9 range
> 인덱스를 하나의 값이 아니라 범위로 검색
- 주로 "<, >, IS NULL, BETWEEN, IN, LIKE"등의 연산자를 통해 인덱스를 검색할 때 사용됨
- 일반적으로 const, ref, range 세가지 접근 방법을 구분하지 않고 묶어서 인덱스 레인지 스캔 또는 레인지 스캔으로 지칭하며,  
인덱스를 효율적으로 사용(작업 범위 결정 조건으로 인덱스를 사용)한다는 표현 모두 이 세가지 접근 방법을 뜻함

#### 10.3.5.10 index_merge
- 2개 이상의 인덱스를 이용해 각각의 검색 결과를 만들어낸 후 결과를 병합해서 처리하는 방식
- 여러 인덱스를 읽어야 하므로 일반적으로 range 접근 방법보다 효율성 낮음
- 전문 검색 인덱스를 사용하는 쿼리에서는 index_merge가 적용되지 않음
- 교집합, 합집합 또는 중복 제거와 같은 부가 작업 필요

#### 10.3.5.11 index
- 인덱스를 처음부터 끝까지 읽는 인덱스 풀 스캔
- index 접근 방법 사용 조건(첫번째+두번째/첫번째+세번째 조건을 충족하는 쿼리에서 사용됨)
  - range나 const, ref 같은 접근 방법으로 인덱스를 사용하지 못하는 경우
  - 인덱스에 포함된 컬럼만으로 처리할 수 있는 쿼리인 경우
  - 인덱스를 이용해 정렬이나 그루핑 작업이 가능한 경우(즉, 별도의 정렬 작업을 피할 수 있는 경우)
    - ```sql
      SELECT * FROM departments ORDER BY dept_name DESC LIMIT 10;

#### 10.3.5.12 ALL
> 풀 테이블 스캔 접근 방법

### 10.3.6 possible_keys 컬럼
- 옵티마이저가 최적의 실행 계획을 만들기 위해 후보로 선정했던 접근 방법에서 사용되는 인덱스 목록(사용될 법했던 인덱스 목록)

### 10.3.7 key 컬럼
- 최종 선택된 실행 계획에서 사용하는 인덱스
- 실행 계획 type이 ALL일 때와 같이 인덱스를 전혀 사용하지 못하면 NULL로 표시됨

### 10.3.8 key_len 컬럼
- 인덱스의 각 레코드에서 몇바이트까지 사용했는지 알려주는 값
- 쿼리를 처리하기 위해 다중 컬럼 인덱스에서 몇개의 컬럼까지 사용했는지 알 수 있음
- 실제 utf8mb4 문자 집합에서 문자 하나가 차지하는 공간은 1~4바이트까지 가변적이지만, MySQL 서버가 utf8mb4 문자를 위해 메모리 공간을 할당해야 할 때는 문자와 관계없이 고정 4바이트로 계산함
- MySQL에서는 Nullable 컬럼에 대해 NULL 여부를 저장하기 위한 1바이트를 추가로 사용

### 10.3.9 ref 컬럼
- 참조 조건(Equal 비교 조건)으로 어떤 값이 제공됐는지 보여줌
- 상숫값을 지정했다면 ref 컬럼의 값은 const로 표시되고, 다른 테이블의 컬럼 값이면 해당 테이블명과 컬럼명이 표시됨

### 10.3.10 rows 컬럼
- rows 컬럼 값은 쿼리를 처리하기 위해 얼마나 많은 레코드를 읽고 체크해야 하는지를 나타냄
- 옵티마이저가 통계 정보를 참조하여 산출한 예상 값이므로 틀릴 가능성이 높으나
  대략의 수치에는 어느정도 근접해야만 제대로 된 실행 계획을 수립할 수 있음(인덱스되지 않은 컬럼이나 컬럼 값이 균등하게 분포되지 않은 경우 제대로 예측을 못할 수 있음)

### 10.3.11 filtered 컬럼
- filtered 컬럼 값은 필터링되고 남은 레코드 예측 비율
- 대부분 쿼리에서 WHERE 절에 사용되는 조건이 모두 인덱스를 사용할 수 있는 것은 아님
- 옵티마이저는 각 테이블에서 일치하는 레코드 개수를 가능하면 정확히 파악해야 좀 더 효율적인 실행 계획 수립

### 10.3.12 Extra 컬럼
- 컬럼 이름과 달리 쿼리 실행 계획에서 성능에 관련된 중요한 내용이 자주 표시됨

#### 10.3.12.1 const row not found
- const 접근 방법으로 테이블을 읽었지만 실제로 해당 테이블에 레코드가 1건도 존재하지 않음

#### 10.3.12.2 Deleting all rows
- 테이블의 모든 레코드를 삭제하고자 한다면 TRUNCATE TABLE 명령을 사용할 것을 권장

#### 10.3.12.3 Distinct
- DISTINCT를 처리하기 위해 조인하지 않아도 되는 행은 모두 무시하고 꼭 필요한 레코드만 읽어서 조인했음

#### 10.3.12.4 FirstMatch
- 세미 조인 최적화 중 FirstMatch 전략이 사용되면 Extra 컬럼에 "FirstMatch(table_name)"
- 테이블명은 기준 테이블을 의미

#### 10.3.12.5 Full scan on Null key
- "col1 IN/NOT IN (SELECT col2 FROM ...)"과 같은 조건을 가진 쿼리에서 발생할 수 있음
    - col1 값이 NULL이 된다면 조건은 "NULL IN (SELECT col2 FROM ...)"과 같이 바뀜
- "Full scan on Null key"는 col1이 NULL 값을 만나면 차선책으로 서브쿼리 테이블에 대해 풀 스캔을 사용할 것이라는 사실을 알려주는 키워드
- col1이 NOT NULL로 정의된 컬럼이면 해당 사항 없음
- 컬럼이 NOT NULL로 정의되진 않았지만 NULL 비교 규칙을 무시해도 된다면 쿼리에 "col1 IS NOT NULL" 조건 지정

#### 10.3.12.6 Impossible HAVING
- HAVING 절의 조건을 만족하는 레코드가 없음

#### 10.3.12.7 Impossible WHERE
- "Impossible HAVING"과 비슷하게 WHERE 조건이 항상 FALSE가 될 수밖에 없는 경우

#### 10.3.12.8 LooseScan
- 세미 조인 최적화 중 LooseScan 최적화 전략이 사용됨

#### 10.3.12.9 No matching min/max row
- MIN()/MAX()와 같은 집합 함수가 있는 쿼리의 조건절에 일치하는 레코드가 한건도 없음

#### 10.3.12.10 no matching row in const table
- 조인에 사용된 테이블에서 const 방법으로 접근할 때 일치하는 레코드가 없다면 표시됨

#### 10.3.12.11 No matching rows after partition pruning
- 파티션된 테이블에 대한 UPDATE/DELETE 명령 대상 파티션이 없음

#### 10.3.12.12 No tables used
- FROM 절이 없는 쿼리나 "FROM DUAL" 형태의 쿼리에 해당

#### 10.3.12.13 NOT exists
- 아우터 조인을 통해 안티-조인(Anti-JOIN)을 수행하는 쿼리에서 각 행에 대해 조인 조건에 일치하는 레코드가 여러 건이 있더라도 딱 1건만 조회해보고 처리하는 최적화를 의미함

#### 10.3.12.14 Plan isn't ready yet
- EXPLAIN FOR CONNECTION 명령이 대상 커넥션에서 아직 쿼리 실행 계획을 수립하지 못한 상태에서 실행됨

#### 10.3.12.15 Range checked for each record(index map: N)
- 매 레코드 별로 인덱스 레인지 스캔할지 풀 테이블 스캔할지 결정하며 처리됨  
  ```sql
  SELECT * FROM employees e1, employees e2 WHERE e2.emp_no >= e1.emp_no;

#### 10.3.12.16 Recursive
- 재귀 CTE(Recursive Common Table Expression) 쿼리 실행 계획에서 출력됨
  ```sql
  WITH RECURSIVE cte (n) AS
  (
    SELECT 1
    UNION ALL
    SELECT n + 1 FROM cte WHERE n < 5
  )
  SELECT * FROM cte;

#### 10.3.12.17 Rematerialize
- 래터럴로 조인되는 테이블은 선행 테이블의 레코드마다 서브쿼리를 실행해서 그 결과를 임시 테이블에 저장함
  이렇게 매번 임시 테이블이 새로 생성되는 경우 "Rematerialize"가 표시됨  
  ```sql
  SELECT * FROM employees e
           LEFT JOIN LATERAL (SELECT *
                              FROM salaries s
                              WHERE s.emp_no = e.emp_no
                              ORDER BY s.from_date DESC LIMIT 2) s2 ON s2.emp_no = e.emp_no
  WHERE e.first_name = 'Matt';
